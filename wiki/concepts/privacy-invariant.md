The single non-negotiable rule that defines this product. Every architectural decision must be checkable against it, and every code path must be auditable for compliance.

## The rule

**Conversation content stays on the device. Always.**

"Conversation content" includes:

- The user's typed messages
- Antoine's generated responses
- Any image the user attaches to a message
- Voice transcripts (when push-to-talk ships)
- System prompts injected into the model context
- Token-stream intermediates
- Anything derived from any of the above (summaries, embeddings, recipe extracts)

What CAN flow to the backend:

- User identity (`AuthUser` shape — email, name, photo URL, subscription state)
- Conversation **metadata** — id, created_at, updated_at, message_count, device_id, sync_status
- Subscription telemetry (purchase token verification, plan tier)
- Crash reports that don't touch any of the above (currently nothing — no SDK installed)

If a column or payload could contain user-generated text from a conversation → it belongs in SQLite only.

## How the rule is enforced

The rule is enforced **by code path**, not by configuration. Each enforcement point must be auditable.

### 1. Inference is on-device

`src/services/inferenceService.ts` is the only thing that constructs the message array (system prompt + history + user turn) and runs inference. It makes ZERO network calls. The model file (`gemma-4-e4b-it.Q4_K_M.gguf`) is loaded from local disk; tokens are generated by `llama.rn` running natively. No remote API.

When the v1 stub is replaced with real `llama.rn`, this property must hold. A code review must confirm zero network imports from this file.

### 2. Persistence is local SQLite only

`src/store/conversationStore.ts` writes to `expo-sqlite` via Drizzle queries in `src/db/queries/`. There is NO sync service. The `is_synced` and `synced_at` columns on `ckm_conversation` exist for a future metadata sync — content never goes through that path.

When the future metadata sync ships (P2 in `tasks/todo.md`), the sync service must be code-reviewed to confirm it touches ONLY the metadata columns, never `content` from `ckm_message`.

### 3. Auth tokens in SecureStore, never in conversation tables

`expo-secure-store` is the only auth-token store. `AsyncStorage` is not installed. Tokens never get logged or appear in error messages.

### 4. Allowlisted hosts for native downloads

The background download module's host allowlist (`pub-7a835c8f4b344301811de8e23b8b3983.r2.dev`) prevents an attacker who controls the backend from tricking the worker into uploading bytes to an arbitrary host. Enforced at TWO layers (JS pre-flight + Kotlin SSRF guard).

### 5. No third-party SDKs that touch conversations

- No analytics SDK installed.
- No crash reporter that uploads stack frames + variables (would leak prompt content if a crash happened during inference).
- Before adding any new dependency, audit its network behavior. If a dependency update introduces remote telemetry on conversation data, revert it.

### 6. Backend schema rule

The web backend may store conversation METADATA. It MUST NOT store conversation content. Before adding any column to a backend table, the discipline (per CLAUDE.md § "Database Design") is to state explicitly: _"This column does not contain conversation content."_

## Why this matters

The audience is culinary professionals — line cooks, restaurateurs, patissiers. Their kitchen IP (recipes, plating, technique, supplier relationships) is their livelihood. The credible promise of "Antoine answers your questions and nothing leaves your phone" only holds if the architecture makes leakage impossible, not just unlikely.

A single SDK that exfiltrates one prompt — even unintentionally, even for "improving the model" — destroys the trust this product is built on.

## Risks + things to watch for

- **Crash reporters.** Sentry, Bugsnag, and similar all want to capture local variables and stack traces. If a crash happens inside `inferenceService` during token generation, the prompt + partial completion would be in the stack frame's locals. Don't install one. If forced to, scrub aggressively or use crash-only-no-context mode.
- **Analytics SDKs.** Mixpanel, Amplitude, etc. typically support "event properties" that developers casually attach. A team member adding `track('message_sent', { content: msg })` would silently leak content. The defense is to not install one.
- **Backup APIs.** Android's auto-backup can include app data. We use `secure_store_backup_rules` and `secure_store_data_extraction_rules` (both XML in `android/app/src/main/res/xml/`) to exclude SecureStore. Confirm SQLite is also excluded before any backup-API work.
- **Future image input.** Antoine has an mmproj projector for multimodal input. When image input ships, the image bytes themselves are conversation content — they cannot be uploaded for "moderation" or "quality improvement."
- **Future voice (P3).** Push-to-talk via `expo-audio` + on-device STT. The STT must be on-device. Cloud STT (Google, Apple, AWS) is forbidden.

## How to audit

When in doubt:

1. `grep -r "fetch\|axios\|http" src/services/inferenceService.ts` — must return zero matches.
2. `grep -r "content" src/services/syncService.ts` (when it exists) — must only appear in column-list constants for metadata, never in payload construction.
3. Check `package.json` deps quarterly for any new analytics/crash-reporting SDK that snuck in.
